# Load test — what's in `/audit` and why

## TL;DR

A subset of the entries visible at [`/audit`](https://regrag.vercel.app/audit) are synthetic — generated by [`scripts/loadtest.py`](../scripts/loadtest.py) on 2026-05-09 to fill the audit log with realistic-looking traffic for demo purposes. They are clearly labeled with `user_id` values starting with `loadtest-` (e.g. `loadtest-07`). If you want to see only real traffic, mentally filter those out, or filter via the API:

```
GET /audit?user_id=eval-runner    # eval-driven invocations
GET /audit                         # everything; demo + eval + synthetic
```

Nothing about this is hidden. The script is checked into the repo. The user_id naming convention makes synthetic rows visually distinguishable in the table.

---

## Why we did this

The audit log is the case study's load-bearing artifact for §5 (Hallucination Control and Auditability). The argument there is "every interaction is logged with the retrieved chunks, the prompt sent, and the model used" — and the schema does capture all of that. But a reviewer who clicks `/audit` and sees ~40 mostly-identical entries from eval runs forms a different impression than what the schema promises. They see the demo's surface, not the design intent.

The load test fills the gap between the schema's capability and the demo's visible state. After the run, `/audit` shows:

- ~80 distinct queries spread across ~20 simulated users
- Mix of single-doc / multi-doc / OOS classifications
- Realistic latency distribution
- Mix of refusal / answer outcomes
- Varying citation-strip counts

A reviewer clicking through gets a tactile sense of what a low-volume production deployment of this system would actually look like, which is the point of having the audit log surface at all.

## What was actually run

```
python scripts/loadtest.py \
    --target https://regrag-api-production.up.railway.app \
    --limit 80 \
    --delay-min 3 --delay-max 10 \
    --paraphrases-per-seed 2 \
    --n-users 20
```

Query bank construction:
1. Took the 40-question eval set; kept the 32 in-scope ones (skipped OOS — paraphrasing OOS questions risks losing the OOS property)
2. Generated 2 paraphrases of each via Sonnet (~64 paraphrases) with a system prompt asking for natural variation — terse, elaborate, abbreviations, indirect references, occasional typos
3. Mixed in 12 hand-authored organic queries from `scripts/loadtest.py` `ORGANIC_QUERIES` — things like "RM18-9 timeline", "ferc 5 minute settlement", "what's the latest on FERC's interconnection reform"
4. Shuffled and capped at 80

Each request used `user_id="loadtest-NN"` where NN was a random integer 1-20 (simulating ~20 distinct demo users), with 3-10 sec random delays between requests.

Total cost: ~$1 in Anthropic spend for the 80 chat invocations (plus ~$0.05 for the paraphrase generation step). Total wall-clock: ~25-30 minutes.

## What it is NOT

- **Not load testing for performance.** Concurrency is 1; this isn't a stress test. It's just realistic-looking traffic generation.
- **Not a benchmark.** Use [`docs/eval-results.md`](eval-results.md) for measured quality. The load test doesn't score outputs against ground truth.
- **Not real users.** Anyone evaluating this for actual operational evidence should disregard the `loadtest-*` user_ids.
